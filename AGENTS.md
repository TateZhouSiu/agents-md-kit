# AGENTS.md

- 默认使用中文与用户沟通；`agent_memory/` 下文档默认使用中文。
- 代码、变量名、接口名、目录名及技术标识遵循仓库原有风格，不因中文沟通而改动。

- 非简单任务先读相关文件，再决定改动方案，不要直接开改。
- 若任务涉及多文件、运行时逻辑、公开接口、复杂排查或预计跨多轮执行，先给出简短计划，再开始修改。
- 非简单任务需自行判断是否适合使用 `multi-agent` skill；当任务涉及并行探索、角色化拆分、审查验证、交接或明确子代理协作时使用。使用 skill 不等于必须启动子代理，是否实际委派仍遵循当前环境和上层规则。
- 对多步骤或长期任务，优先将需求转成可验证的成功标准、范围边界、停止条件和验证方式；使用 `/goal` 或等价长期目标模式时也必须遵循，遇到权限、范围、产品决策或风险不明确时停止并汇报。
- 默认最小改动，不做无关重构，不扩大改动范围。
- 优先复用现有实现、组件、工具和模式。
- 每次改动都应能追溯到用户请求、已确认假设或必要验证；发现无关问题只记录风险，不顺手重构或扩大范围。

- 使用 `agent_memory/context.md` 维护项目级上下文，使用 `agent_memory/progress.md` 维护当前任务进度，使用 `agent_memory/bugs.md` 维护问题与风险记录。
- 若当前项目根目录缺少 `agent_memory/`、`agent_memory/context.md`、`agent_memory/progress.md`、`agent_memory/bugs.md` 或 `agent_memory/archive/`，则在开始非简单任务前先补齐；`agent_memory/archive/` 只创建目录，三个 `.md` 文件必须按 `~/.codex/templates/agent_memory/` 下对应模板原样创建，不要改写模板结构。
- 非简单任务开始前先读取这三个文件；出现阶段完成、方向变化、失败尝试、准备结束会话时，必须更新相关文件。
- 这三个文件只保留当前有效信息，不写成无限追加日志。
- 文件过长时先压缩为摘要，再归档旧内容到 `agent_memory/archive/`。

- 若改动涉及运行时代码、构建行为、测试、类型、公开接口或关键 UI 行为，必须执行与改动范围匹配的验证。
- 优先运行最小必要验证，并明确说明已验证和未验证部分。

- 若需求有歧义，优先采用最保守、最小范围、最容易验证的方案，并说明假设。
- 不要默默选择有风险的解释；若存在多个合理理解，先说明假设、权衡和建议方案，必要时询问用户。
- 不要把猜测当结论；未确认的内容必须明确标注。

- 同类问题若重复出现，再把经验固化进本文件。
- 若某类流程反复出现，不要继续堆到 `AGENTS.md`，优先升级为独立 skill、hook 或子代理。
- 保持本文件简短、稳定、可执行。
